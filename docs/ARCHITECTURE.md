# Architecture

How the Oasis Health Services website is structured and built. See [CLAUDE.md](../CLAUDE.md) for the quick reference and [CONVENTIONS.md](CONVENTIONS.md) for coding style.

## Rendering model: Astro SSG + React islands

The site is statically generated by **Astro 5**. Every route is pre-rendered to HTML at build time; there is no application server and no client-side router. Interactivity is added with the **islands** pattern: a `.astro` page renders the static shell and embeds React components that hydrate in the browser.

### When to use `.astro` vs React

- **`.astro`** — page shells, layouts, SEO/meta, and content that does not need browser-side state. This is the default for anything static.
- **React (`.tsx`/`.jsx`)** — anything with `useState`/`useEffect`, event handlers, forms, dialogs, or other client interactivity. React components are mounted inside `.astro` files with a `client:*` directive.

### Hydration directives

| Directive | Use for | Examples |
|-----------|---------|----------|
| `client:load` | Above-the-fold / immediately-interactive UI. The dominant pattern. | `Header` and `CookieConsent` in [src/layouts/Layout.astro](../src/layouts/Layout.astro); the homepage island in [src/pages/index.astro](../src/pages/index.astro); the multi-step intake in [src/pages/start.astro](../src/pages/start.astro) |
| `client:visible` | Below-the-fold secondary sections that can hydrate lazily. | FAQ / secondary blocks in the Help Center request pages, e.g. [src/pages/resources/help-center/request/prescription.astro](../src/pages/resources/help-center/request/prescription.astro) |

Prefer the narrowest directive that still works — reach for `client:visible` for below-the-fold content rather than hydrating everything on load.

```
src/
├── pages/        # File-based routes (.astro). e.g. services/[slug].astro, patients/verify-insurance.astro
├── layouts/      # Astro layouts: Layout, BaseLayout, BlogLayout, LandingLayout
├── components/
│   ├── common/   # Header, Footer, CookieConsent, CrisisNotice, shared chrome
│   ├── ui/       # Radix-based primitives (lowercase filenames: button.tsx, card.tsx, select.tsx)
│   ├── forms/    # Form framework + reusable field components (RHF + Zod)
│   ├── pages/    # Page-level React components (Home, ContactUs, etc.)
│   ├── blog/     # Blog UI (ArticleCard, BlogSidebar, ...)
│   └── lp/       # Landing-page sections (Hero, Features, ...)
├── content/      # Astro content collections: blog/, authors/, categories/
├── lib/          # Typed data (*-data.ts), Zod schemas, utils, Firebase, assessments
├── store/        # Zustand stores (form/UI state)
├── api/          # Firebase-backed submission/upload handlers
└── styles/       # global.css (design tokens), blog.css, lp.css

tools/            # Build pipeline: pipeline.js, generate-webp.js, build-seo.js, setup-staging-auth.js
public/           # Static assets, robots.txt, llms.txt, images/
.github/workflows # CI/CD (dev CI, staging/prod PR checks, manual deploy)
```

## Data layer

Structured content is kept as **typed TypeScript datasets** in `src/lib/*-data.ts` and imported directly by pages and components — there is no runtime database for this content.

- Each dataset exports an interface plus an array, e.g. the `Provider` interface and `providers` array in [src/lib/providers-data.ts](../src/lib/providers-data.ts). Other datasets: `services-data.ts`, `conditions-data.ts`, `insurance-data.ts`.
- **Blog/authors/categories** are managed as Astro **content collections** under [src/content/](../src/content/) (markdown + frontmatter), validated by the collection schema.
- Clinical self-assessments (PHQ-9, GAD-7, ASRS, PC-PTSD-5) live as typed modules under `src/lib/assessments/`.

## Forms & state

Multi-step intake and Help Center forms follow a consistent pattern:

- **Validation**: Zod schemas in [src/lib/schema.ts](../src/lib/schema.ts) (e.g. `ContactSchema`, `InsuranceSchema`, `IntakeFormSchema`), wired into React Hook Form via `@hookform/resolvers`.
- **Reusable fields**: generic, controlled field components in [src/components/forms/](../src/components/forms/) (`SelectField`, `CheckboxField`, `RadioGroupField`, `ContactFields`, `InsuranceFields`, `FileUploadZone`, …) and the `MultistepForm` framework.
- **Shared state**: `useFormStore` in [src/store/index.ts](../src/store/index.ts) tracks `currentStep`, `isSubmitting`, `submitted`, and uploaded `files`, with `next`/`prev`/`reset` actions.
- **Submission**: handled through Firebase in [src/api/](../src/api/); App Check (reCAPTCHA Enterprise) guards submissions.

## Build & deploy pipeline

`npm run build` runs [tools/pipeline.js](../tools/pipeline.js), which executes stages in order:

1. **Image optimization** — `tools/generate-webp.js` recursively converts images to WebP (runs in every environment, and also before `astro dev`).
2. **Astro build** — `astro build`, adding `--mode staging` and `--site $PUBLIC_SITE_URL` when set.
3. **SEO generation** — `tools/build-seo.js` (sitemaps, `robots.txt`, `llms.txt`, schema). **Runs for staging & production only**, not a plain local build.
4. **Staging auth** — `tools/setup-staging-auth.js` injects basic auth (**staging only**).

### Environments & deployment

- Env files: `.env.local` (default `build`), `.env.staging`, `.env.production`, selected via Node's `--env-file`.
- Deploy is rsync of `dist/` to the host: `deploy:staging` → `dev.oasishealthservices.com`, `deploy:prod` → `oasishealthservices.com`. `*-test` variants do an rsync `--dry-run`.
- CI/CD lives in `.github/workflows/` (dev CI, staging/prod PR checks, manual deploy trigger).

For SEO specifics (meta tags, structured data, GTM, sitemap details) see [SEO-IMPLEMENTATION.md](../SEO-IMPLEMENTATION.md).
